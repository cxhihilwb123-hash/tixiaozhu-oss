# Contributor Rewards

Tixiaozhu values real, useful contributions. This page explains how contributors can be recognized and, when available, receive small thank-you rewards.

This is not a paid job board, a star-for-reward campaign, or a guarantee of payment. Rewards are meant to encourage honest review, high-quality pull requests, and long-term maintainership.

## What Counts

Eligible contributions include:

- Running the Quick Start and reporting clear setup problems.
- Opening a well-scoped issue with evidence, screenshots, logs, or reproduction steps.
- Reviewing the education-product flow from a parent, teacher, student, or operator perspective.
- Improving README, API docs, deployment docs, or contributor onboarding.
- Submitting a pull request that passes CI and improves product quality.
- Adding or improving tests, audits, accessibility coverage, or runtime-safety checks.
- Improving question-bank quality checks or teaching-content structure.
- Helping another contributor understand setup, labels, issues, or project intent.

## Reward Levels

### Level 1: Public Thanks

For helpful issues, comments, reviews, or small documentation improvements.

Possible recognition:

- Thank-you comment from the maintainer.
- Mention in a release note or changelog when the contribution affects a release.
- Contributor listing in future project documentation.

### Level 2: Contributor Highlight

For merged pull requests, useful review threads, or repeated high-signal feedback.

Possible recognition:

- Highlight in a GitHub Discussion update.
- Mention in a release note.
- Invitation to review follow-up issues before they are prioritized.

### Level 3: Maintainer Track

For repeated, calm, useful participation over time.

Possible recognition:

- Maintainer-candidate mention.
- More direct coordination on roadmap issues.
- Optional invitation to help triage issues or review small pull requests.

### Optional Thank-You Gifts

The maintainer may occasionally offer small thank-you gifts, credits, or bounties for clearly scoped work. These are optional and must be confirmed in writing before the work starts.

If an issue has a reward, it should state:

- Reward type.
- Acceptance criteria.
- Review process.
- Whether the reward is first-come, maintainer-selected, or split across contributors.

No reward is owed unless the maintainer explicitly confirms it on the issue or in another written channel.

## What Does Not Count

The project does not reward:

- Fake stars, fake praise, fake reviews, or fake usage claims.
- Spam comments, low-effort AI-generated issues, or duplicate reports.
- Private data, leaked credentials, or screenshots containing sensitive information.
- Security reports posted publicly instead of following `SECURITY.md`.
- Pull requests that ignore review feedback or bypass CI.
- Contributions copied from other projects without respecting their license.

## How To Claim Recognition

Comment on the relevant issue or pull request with a short note:

```text
Contribution summary:

- What I changed or reviewed:
- Evidence or link:
- Which reward level I think this fits:
```

The maintainer makes the final decision. When in doubt, ask before starting large work.

## Fairness Notes

- Rewards are based on usefulness, not popularity.
- Small contributions are welcome.
- Honest criticism is welcome.
- Maintainer decisions should be transparent and tied to project needs.
- This policy may change as the project grows.
