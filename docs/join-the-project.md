# Join the Project

Tixiaozhu welcomes practical contributors, reviewers, and educators who want to make this AGPL education-product sample easier to run, inspect, and improve.

Repository:

```text
https://github.com/cxhihilwb123-hash/tixiaozhu-oss
```

## Good Ways To Help

- Run the Quick Start and report anything confusing.
- Review the screenshots and product flows from an education or parent perspective.
- Improve setup, API, deployment, or PostgreSQL documentation.
- Add focused UI accessibility checks.
- Add tests around student/admin flows, auth scoping, payment deferral, and OCR deferral.
- Improve question-bank content quality checks.
- Refactor backend route and service code in small behavior-preserving pull requests.

## Contributor Tracks

### First-Time Contributor

Best starting point:

1. Clone the repository.
2. Follow the Quick Start in `README.md`.
3. Open an issue with one clear setup, docs, or UX improvement.
4. Pick a `good first issue` only after you can run the project locally.

Useful labels:

- `good first issue`
- `help wanted`
- `docs`
- `question-bank`

### Engineer Reviewer

Helpful review areas:

- API clarity and route organization.
- CI gates and audit scripts.
- Security posture around sessions, auth, CORS, payment deferral, and production defaults.
- Whether the project is easy to clone, run, test, and reason about.

### Education Reviewer

Helpful review areas:

- Whether the practice flows make sense for primary-school learners.
- Whether the question-bank structure is understandable.
- Whether the product language is clear for parents, teachers, and operators.
- Which subject, grade, or content-quality gaps matter most.

### Maintainer Candidate

Maintainer help is welcome after repeated useful participation. A good maintainer candidate usually:

- Opens specific, evidence-based issues.
- Sends small pull requests that pass CI.
- Responds calmly to review.
- Helps other contributors understand setup or project intent.
- Respects the AGPL license and commercial licensing boundaries.

## How To Ask To Participate

Open a GitHub issue using the "Contributor interest" template, or comment on a `help wanted` issue with:

```text
I can help with:

- Area:
- Time available:
- Relevant experience:
- First task I want to try:
```

No formal application is needed for small contributions. Pull requests and honest issues are the best introduction.

## Ground Rules

- Honest critique is more useful than praise.
- Do not claim production usage unless you are actually using the project.
- Do not submit private student data, private screenshots, secrets, or local `.env` files.
- Keep first pull requests small.
- Security issues should follow `SECURITY.md` instead of being posted publicly.
